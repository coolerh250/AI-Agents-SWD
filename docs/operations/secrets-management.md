# Secrets Management (Stage 26)

This runbook covers the Stage 26 external-secrets baseline. The platform
ships three SecretProvider classes behind one interface; this document
explains the inventory, the rotation contract, and the leak-scan
procedure operators run before promoting any change.

## Secret inventory

Single source of truth: `infra/runtime/secrets.inventory.yml`. Run

```
./scripts/list_required_secrets.py
./scripts/list_required_secrets.py --env staging
./scripts/list_required_secrets.py --json
```

to print the per-secret summary. The script validates the file's
structure (every entry carries `name`, `required_for`, `environments`,
`provider`, `rotation_policy`, `leak_risk`, `allowed_to_be_missing_in_*`)
and exits non-zero on a structural failure. **It never prints a secret
value.**

Required names today:

* `POSTGRES_PASSWORD` â€” staging + production
* `GITHUB_TOKEN` â€” opt-in only
* `DISCORD_BOT_TOKEN`, `DISCORD_TEST_CHANNEL_ID` â€” opt-in only
* `ALERTMANAGER_WEBHOOK_URL` â€” staging null-receiver tolerated;
  production-check requires it
* `VAULT_ADDR`, `VAULT_TOKEN` â€” required when `SECRET_PROVIDER=vault`

## Providers

```python
from shared.sdk.secrets import provider_from_env, SecretRef

provider = provider_from_env()                    # SECRET_PROVIDER drives selection
ref = provider.get_secret("GITHUB_TOKEN")         # SecretRef
if ref:                                           # __bool__ â€” present and non-empty
    headers["Authorization"] = f"Bearer {ref.reveal()}"
```

### EnvSecretProvider â€” `SECRET_PROVIDER=env`

Reads `os.environ`. The default for local/test. Production-check rejects
`env` as the sole provider (operators must wire Vault).

### VaultKvSecretProvider â€” `SECRET_PROVIDER=vault`

Reads Vault KV v2 over HTTP at
`${VAULT_ADDR}/v1/${VAULT_KV_MOUNT}/data/${VAULT_KV_PATH}`. Defaults:

* `VAULT_KV_MOUNT=secret`
* `VAULT_KV_PATH=aiagents/staging`

The Vault token is held in a `SecretRef`; the provider never echoes it
to logs, the status view, or any HTTP response. A failed Vault read
returns `SecretRef(present=False)` so callers fall back to their safe
path; the failure reason is exposed via `provider.status["last_error"]`
(string, never the token).

### MockVaultSecretProvider â€” `SECRET_PROVIDER=mock-vault`

Reads a flat JSON file at `MOCK_VAULT_SECRETS_FILE`
(default `infra/runtime/.mock-vault-secrets.local.json`). The file is
generated by `scripts/bootstrap_mock_vault_secrets.sh`, chmod `600`,
gitignored. The provider refuses to return values matching common
real-token prefixes unless `MOCK_VAULT_ALLOW_REAL_TOKEN_SHAPES=true`.

**This provider is for staging validation only.** Production-check
refuses it.

## Token redaction policy

* `SecretRef.__repr__()` and `__str__()` return `***REDACTED***`.
* `SecretRef.model_dump()` returns `{name, present, value="***REDACTED***"}`.
* Provider `status` dicts contain booleans + provider/mount/path strings.
  **No status field carries a secret value.**
* `redact_mapping()` rewrites any key matching `token` / `secret` /
  `password` / `passwd` / `api_key` / `apikey` / `credential` /
  `private_key` to the redaction token.

## Rotation smoke

```
./scripts/verify_secret_rotation_smoke.sh
```

Drives the SDK-level rotation contract:

1. Write version A into a temp mock-vault file.
2. Build a `MockVaultSecretProvider`; read A.
3. Rewrite the file with version B.
4. Call `provider.reload()`; read B.
5. Assert `A != B` and that the provider status carries no value.

Service-layer rotation is currently a documented limitation â€” services
cache the SecretRef at process boot. A future stage may add a
`/operations/secrets/reload` endpoint; today the rotation procedure is
to restart the affected service.

## Leak scan

```
./scripts/scan_for_secret_leaks.sh
```

Scans `README.md`, `docs/`, `source/progress.md`, the runtime-health
logs, the runtime env templates, the compose files, and `scripts/` for
substrings matching:

* `ghp_[A-Za-z0-9_]{16,}`
* `github_pat_[A-Za-z0-9_]{16,}`
* `xoxb-[A-Za-z0-9-]{8,}`
* `Bot [A-Za-z0-9._-]{20,}`
* `Bearer [A-Za-z0-9._-]{30,}`
* `sk-[A-Za-z0-9]{20,}`

Allowed: `PLACEHOLDER_DO_NOT_COMMIT_REAL_VALUE`, `***REDACTED***`,
`${GITHUB_TOKENâ€¦}` / `${DISCORD_BOT_TOKENâ€¦}` / `${VAULT_TOKENâ€¦}` /
`${POSTGRES_PASSWORDâ€¦}` env-var references, and the scanner's own
regex docstrings.

Exit code 0 -> `SECRET_LEAK_SCAN: PASS`. Failure prints the
file:line + matched pattern name, never the matched substring.

## Production restrictions

`validate_runtime_config.py --mode production-check` refuses:

* `SECRET_PROVIDER=mock-vault`
* `SECRET_PROVIDER=env` (env-only is acceptable for local but not for
  production)
* `SECRET_PROVIDER=vault` without `VAULT_ADDR` + `VAULT_TOKEN`
* `VAULT_TOKEN=PLACEHOLDER_DO_NOT_COMMIT_REAL_VALUE`
* Vault `server -dev` address (`http://vault:8200`,
  `http://localhost:8200`)
* `ALERTMANAGER_WEBHOOK_URL` empty / placeholder
* `PRODUCTION_EXECUTED_TRUE_COUNT != 0`

## End-to-end staging verification

```
./scripts/verify_staging_secrets.sh
```

Runs the full sequence:

1. `list_required_secrets.py`
2. `bootstrap_mock_vault_secrets.sh`
3. `validate_runtime_config.sh --mode staging` (SECRET_PROVIDER=mock-vault)
4. `validate_runtime_config.sh --mode production-check` refuses
   mock-vault
5. `verify_secret_rotation_smoke.sh`
6. `scan_for_secret_leaks.sh`
7. `start_staging_runtime.sh` with `SECRET_PROVIDER=mock-vault`
8. `GET /operations/safety` carries the Stage 26 fields
9. Real GitHub / Discord still disabled
10. `stop_staging_runtime.sh` (default `--down`)

The verifier defaults to tear-down after PASS; pass `--keep-running`
to inspect the staging stack.
